# ТЗ для Василия: web-версия VK/TG

## Роль
Василий отвечает за единый frontend, который управляет генерацией и публикацией постов в VK и Telegram.

## Цель
Сделать один интерфейс, в котором пользователь задает тему и тон, выбирает платформы, получает текст и изображение, а затем публикует результат в выбранные каналы.

## Формат результата
Нужно сделать именно web-версию продукта, которая открывается в браузере и является основной пользовательской оболочкой для работы с VK и Telegram.

## Область ответственности
1. Web-интерфейс генерации контента.
2. Web-интерфейс результата с возможностью ручной правки.
3. Запуск публикации в VK и Telegram из браузера.
4. Понятный вывод статусов успеха и ошибок.
5. Базовая пригодность интерфейса для демонстрации и ежедневной работы.

## Приоритеты задач
### Must
1. Рабочее открытие web-версии по `/`.
2. Форма ввода темы и тона.
3. Выбор платформ VK и Telegram.
4. Отображение статуса сервисов.
5. Генерация контента через backend.
6. Ручная правка текста.
7. Публикация через backend.
8. Раздельный показ статусов по платформам.
9. Сохранение формы после ошибки.

### Should
1. Более удобные тексты подсказок.
2. Чище визуальная иерархия блоков.
3. Более понятные статусы загрузки.

## Этапы выполнения
1. Этап 1. Поднять web-каркас и подключить `/api/status`.
2. Этап 2. Подключить сценарий генерации.
3. Этап 3. Подключить сценарий публикации.
4. Этап 4. Проверить негативные сценарии и поведение после ошибок.
5. Этап 5. Подготовить итоговую демонстрацию для тимлида.

## Статусы выполнения
1. `не начато` - задача еще не взята в работу.
2. `в работе` - задача реализуется.
3. `на проверке` - задача передана на приемку.
4. `принято` - задача утверждена.

5. `Публикация VK/TG` - статус: `на проверке`
6. `Отображение ошибок` - статус: `на проверке`
7. `Адаптивность` - статус: `на проверке`

## Что уже задано тимлидом
1. Web-версия работает поверх общего backend API.
2. Нельзя хардкодить адреса токенов и секретов на клиенте.
3. Интерфейс должен быть один, без разделения на отдельные страницы под VK и Telegram.
4. Поддерживаются API:
   - `GET /api/status`
   - `POST /api/generate`
   - `POST /api/publish`
5. Web-версия должна открываться из backend по `/`.

## Пользовательский сценарий
1. Пользователь вводит тему поста.
2. Пользователь задает тон.
3. Пользователь выбирает VK, Telegram или обе платформы.
4. Пользователь нажимает "Сгенерировать".
5. Пользователь видит текст, промпт картинки и ссылку на изображение.
6. Пользователь при необходимости редактирует текст.
7. Пользователь нажимает "Опубликовать".
8. Интерфейс показывает статус по каждой платформе отдельно.

## Требования к web-версии
1. Веб-страница должна запускаться без отдельного frontend-сервера.
2. Интерфейс должен быть понятным без технических знаний.
3. Пользователь должен видеть доступность OpenAI, VK и Telegram.
4. Пользователь должен иметь возможность вручную исправить текст перед публикацией.
5. Если картинка сгенерирована, нужно показать превью.
6. Если одна платформа упала, web-версия должна показать частичный успех, а не общий сбой.

## Обязательные экраны и блоки
1. Экран или основной блок ввода темы и тона.
2. Блок выбора платформ: VK, Telegram, обе сразу.
3. Блок статуса доступности сервисов.
4. Блок результата генерации с редактируемым текстом.
5. Блок промпта изображения.
6. Блок ссылки на изображение и превью.
7. Блок логов или статусов публикации по платформам.
8. Кнопка генерации и кнопка публикации с понятными состояниями загрузки.

## Нефункциональные требования
1. Web UI должен работать без сборщика, как статические файлы.
2. Ошибки API должны быть читаемыми для пользователя.
3. Нужно отображать состояние доступности сервисов.
4. Нельзя блокировать публикацию, если одна платформа недоступна, а другая доступна.
5. Интерфейс должен быть адаптивным хотя бы на базовом уровне.
6. UI не должен требовать перезагрузки страницы для перехода между шагами сценария.
7. Нельзя скрывать ошибки публикации за общим сообщением "что-то пошло не так".
8. Все действия пользователя должны давать визуальную обратную связь.

## Запрещено
1. Делать web-версию, которая работает только на локальных захардкоженных URL.
2. Прятать критичные статусы интеграций.
3. Блокировать ручную правку сгенерированного текста.
4. Делать отдельный UI только для Telegram или только для VK вместо единого интерфейса.
5. Требовать от пользователя прямой работы с API или консолью.

## UX-критерии приемки
1. Пользователь без инструкции понимает, куда ввести тему и как запустить генерацию.
2. После генерации пользователь сразу видит, что именно будет опубликовано.
3. Если изображение не сгенерировалось, пользователь понимает, можно ли публиковать без него.
4. Если VK и Telegram дали разные результаты, это видно отдельно.
5. Пользователь понимает, завершилась ли операция успешно, частично успешно или с ошибкой.

## Definition of Done
1. Web-версия открывается из backend по `/`.
2. Генерация работает через `/api/generate`.
3. Публикация работает через `/api/publish`.
4. Пользователь видит превью изображения.
5. Пользователь видит раздельные статусы по VK и Telegram.
6. Полный сценарий "сгенерировать -> поправить -> опубликовать" проходит в браузере.
7. Все обязательные блоки интерфейса присутствуют и работают без ручных костылей.
8. Интерфейс не теряет введенный пользователем текст при неуспешной публикации.

## Чеклист приемки web-версии
1. Открытие `/` показывает рабочий интерфейс без ошибок в первом сценарии.
2. Статус сервисов отображается до первой генерации.
3. При генерации без картинки UI не ломается и корректно показывает пустое превью.
4. При успешной генерации кнопка публикации становится доступной.
5. Пользователь может изменить текст перед публикацией.
6. Результат публикации отображается отдельно для VK и Telegram.
7. После ошибки публикации интерфейс остается рабочим и не очищает форму.
8. На ширине мобильного экрана интерфейс остается читаемым.

## Формат отчетности Василия
1. В конце каждой итерации Василий пишет, какие пользовательские сценарии уже закрыты.
2. Отдельно перечисляет, какие состояния ошибок отображаются в UI.
3. Отдельно указывает, какие ограничения UX пока остались.


## Контрольные точки
1. Сначала поднять форму и статус сервисов.
2. Затем подключить генерацию.
3. Затем подключить публикацию.
4. После этого передать мне список UX-ограничений и спорных мест.
