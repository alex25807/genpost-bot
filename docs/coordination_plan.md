# Координация работ

## Тимлид Андрей: решение по архитектуре
1. Базовый проект переводим из набора скриптов в единый пакет.
2. Общая генерация текста и изображения живет в backend.
3. Публикация по платформам разносится в отдельные publisher-модули.
4. Web-версия работает только через HTTP API, без прямых вызовов OpenAI, VK или Telegram.
5. Telegram-версия и web-версия развиваются как части одного продукта, а не как отдельные несвязанные решения.

## Контракт между Сергеем и Василием
1. Сергей не меняет формат ответов API без согласования.
2. Василий не хардкодит обходные схемы под ошибки backend.
3. Общий объект публикации:

```json
{
  "content": "string",
  "image_url": "string|null",
  "platforms": ["vk", "telegram"]
}
```

4. Общий объект результата:

```json
{
  "results": [
    {
      "platform": "vk",
      "success": true,
      "details": {}
    }
  ]
}
```

5. Любое изменение в `status`, `generate`, `publish` сначала фиксируется в документации, потом в коде.
6. Частичный успех публикации считается обязательным поддерживаемым сценарием.

## Жесткие правила координации
1. Нельзя делать локальные договоренности "на словах" без обновления документов.
2. Backend считается готовым только если им реально может пользоваться web-версия.
3. Frontend считается готовым только если проходит сценарий с реальными ответами backend.
4. Любой спор по контракту решается в пользу простого и стабильного JSON.
5. Любой временный костыль должен быть явно помечен как риск.

## Приоритеты продукта
### Must
1. Единый backend API.
2. Рабочая web-версия.
3. Публикация в VK.
4. Публикация в Telegram.
5. Сценарий частичного успеха.
6. Безопасное хранение секретов через `.env`.

### Should
1. Более подробная диагностика ошибок.
2. Более удобный UX для оператора.
3. Улучшенное логирование результатов публикации.

## Этапы проекта
1. Этап A. Аудит текущего VK-решения и фиксация архитектуры.
2. Этап B. Сборка общего backend API.
3. Этап C. Сборка web-версии.
4. Этап D. Встраивание Telegram.
5. Этап E. Интеграционная приемка.

## Управленческие статусы
1. `не начато`
2. `в работе`
3. `на проверке`
4. `принято`

## Сводная таблица статусов
1. `Архитектура` - статус: `принято`
2. `Backend API` - статус: `принято`
3. `Web-версия` - статус: `на проверке`
4. `Telegram-интеграция` - статус: `на проверке`
5. `Интеграционная приемка` - статус: `в работе`

## Контроль качества от тимлида
1. Все ключи и токены только через `.env`.
2. Существующий VK-сценарий не должен потеряться.
3. Единая точка входа запуска: `python main.py`.
4. Для ручной проверки обязателен сценарий: сгенерировать -> отредактировать -> опубликовать.
5. Обязательны две проверяемые поставки: Telegram-функциональность и рабочая web-версия.
6. Нельзя сдавать работу без демонстрации негативных сценариев и отображения ошибок.

## Приемка тимлидом
1. Проверяю, что `/api/status` отражает состояние интеграций.
2. Проверяю, что `/api/generate` не ломает UI при отсутствии картинки.
3. Проверяю, что `/api/publish` корректно показывает успех, частичный успех и ошибку.
4. Проверяю, что web-версия не требует знания API для базовой работы.
5. Проверяю, что секреты не вернулись в кодовую базу.


## Интеграционный результат
1. В проекте есть backend API.
2. В проекте есть рабочая web-версия.
3. В проекте есть публикация в VK и Telegram.
4. Проект можно развернуть из одного набора файлов.
