# ТЗ для Сергея: backend Telegram и web API

## Роль
Сергей отвечает за backend-направление Telegram внутри общего пакета генерации постов.

## Цель
Добавить надежную публикацию в Telegram без дублирования логики генерации текста и изображения. Telegram должен работать не как отдельный скрипт, а как часть общего backend, который обслуживает и Telegram-ветку, и web-версию продукта.

## Область ответственности
1. Поддержка Telegram Bot API в едином backend.
2. Поддержка backend-контрактов для web-версии.
3. Публикация текста и публикация текста с изображением.
4. Корректная обработка ошибок Telegram и backend-ошибок для UI.
5. Совместимость с общими контрактами `/api/status`, `/api/generate`, `/api/publish`.

## Приоритеты задач
### Must
1. Рабочий `GET /api/status`.
2. Рабочий `POST /api/generate`.
3. Рабочий `POST /api/publish`.
4. Публикация Telegram без изображения.
5. Публикация Telegram с изображением.
6. Стабильный JSON-ответ для web-версии.
7. Обработка ошибок без падения UI.

### Should
1. Более подробные сообщения об ошибках Telegram.
2. Улучшенная валидация входных payload.
3. Дополнительные диагностические поля в `details`.

## Этапы выполнения
1. Этап 1. Зафиксировать и проверить backend-контракты.
2. Этап 2. Реализовать Telegram text-only публикацию.
3. Этап 3. Реализовать Telegram text+image публикацию.
4. Этап 4. Проверить совместимость с web-версией.
5. Этап 5. Сдать итоговый отчет и риски.

## Статусы выполнения
1. `не начато` - задача еще не взята в работу.
2. `в работе` - идет реализация.
3. `на проверке` - задача готова и передана тимлиду.
4. `принято` - задача проверена и одобрена.

## Таблица контроля Сергея
1. `GET /api/status` - статус: `принято`
2. `POST /api/generate` - статус: `принято`
3. `POST /api/publish` - статус: `принято`
4. `Telegram text-only` - статус: `на проверке`
5. `Telegram text+image` - статус: `на проверке`
6. `Совместимость с web-версией` - статус: `на проверке`

## Что уже задано тимлидом
1. Общий backend строится вокруг `FastAPI`.
2. Генерация текста и изображения остается общей, а не платформенной.
3. Telegram подключается как отдельный publisher, а не как отдельный скрипт.
4. Web-версия работает только через backend API, прямого доступа к внешним сервисам у клиента нет.
5. Конфиг хранится только через `.env`.

## Технический контракт
1. Вход в публикацию: `content`, `image_url`, `platforms`.
2. Для Telegram используются `TELEGRAM_BOT_TOKEN` и `TELEGRAM_CHAT_ID`.
3. Если есть картинка, используется `sendPhoto`.
4. Если картинки нет, используется `sendMessage`.
5. Для web-версии обязательны endpoints:
   - `GET /api/status`
   - `POST /api/generate`
   - `POST /api/publish`
6. Ответ для frontend должен возвращаться в формате:

```json
{
  "results": [
    {
      "platform": "telegram",
      "success": true,
      "details": {}
    }
  ]
}
```

## Нефункциональные требования
1. Ошибки Telegram не должны падать необработанным traceback в UI.
2. Лимит caption Telegram должен учитываться.
3. Таймауты HTTP-запросов обязательны.
4. API должно быть пригодно для web-версии без дополнительной ручной склейки.
5. Никаких токенов в коде и коммитах.
6. Нельзя ломать существующие контракты API без предварительного согласования с тимлидом.
7. Все ответы API должны быть предсказуемыми по структуре, даже в случае ошибки.
8. Нельзя оставлять заглушки вида "потом доделаю" в ключевых местах публикации и валидации.

## Запрещено
1. Делать отдельный Telegram-скрипт в обход общего backend.
2. Возвращать на frontend сырые traceback или нестабильные ответы разных форматов.
3. Хардкодить токены, chat_id, group_id и ключи OpenAI.
4. Менять JSON-контракт без фиксации изменений в документации.
5. Перекладывать логику генерации на frontend.

## Обязательные backend-сценарии
1. `GET /api/status` корректно показывает доступность OpenAI, VK и Telegram.
2. `POST /api/generate` возвращает текст, список платформ, промпт изображения и ссылку на изображение либо `null`.
3. `POST /api/publish` умеет отработать частичный успех, если одна платформа опубликовала, а вторая нет.
4. Публикация в Telegram без изображения работает отдельно.
5. Публикация в Telegram с изображением работает отдельно.
6. Один и тот же запрос на публикацию должен быть пригоден для вызова из web-версии без доработки payload.

## Definition of Done
1. Telegram-пост публикуется из общего интерфейса.
2. Telegram можно вызывать вместе с VK одним запросом.
3. Web-версия может получить статус сервисов, сгенерировать контент и отправить публикацию только через backend API.
4. Ошибка неверного токена или chat_id возвращается в понятном виде.
5. Код не ломает существующую VK-логику.
6. Все три endpoints задокументированы и совпадают с реальным поведением.
7. Есть ручная проверка как минимум четырех сценариев: генерация, Telegram text-only, Telegram text+image, смешанная публикация VK+Telegram.

## Формат отчетности Сергея
1. В конце каждой итерации Сергей пишет, какие endpoints изменил.
2. Отдельно указывает, менялся ли контракт JSON.
3. Отдельно указывает, какие ошибки обработал.
4. Отдельно пишет, что осталось риском.


## Чеклист приемки
1. При отсутствии `TELEGRAM_BOT_TOKEN` backend не падает при старте приложения.
2. При запросе в Telegram с неверным токеном пользователь получает понятную ошибку в `details.error`.
3. При публикации без картинки вызывается логика text-only.
4. При публикации с картинкой вызывается логика image-post.
5. При отказе Telegram публикация в VK не считается автоматически проваленной.
6. Ответ `POST /api/publish` всегда содержит массив `results`.

## Контрольные точки
1. Сначала проверить контракты `GET /api/status`, `POST /api/generate`, `POST /api/publish`.
2. Затем протестировать `sendMessage`.
3. Затем протестировать `sendPhoto`.
4. Затем проверить, что web-клиент может пройти полный сценарий без прямых вызовов Telegram API.
5. После этого отдать мне короткий отчет: что готово, что не готово, какие риски остались.
